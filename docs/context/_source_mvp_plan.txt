Executive Summary  
MVP Objective  
Build a functional Question Bank and Exam Engine that demonstrates the core value 
proposition of SPECTROPY LMS. The MVP must support the 3 -tier content ownership model 
(Platform, Tenant, School) while delivering a usable product for early adopters.  
What We  Are Building  
• Question Bank: Create, manage, search questions with 3 -tier ownership  
• Exam Engine: Create exams, conduct tests, auto -grade, show results  
• Foundation: Multi -tenant auth, role -based access, basic UI  
What We Are NOT Building (MVP)  
• Courses module (Phase 2)  
• Advanced analytics and reporting (Phase 2)  
• Mobile app (Phase 2)  
• Proctoring features (Phase 2)  
• Bulk import/export (Phase 2)  
• Content entitlement system (simplified for MVP)  
Timeline Overview  
Phase  Focus  Duration  Outcome  
Phase 0  Foundation Setup  1 week  Project structure, DB, Auth  
Phase 1  Question Bank Core  4 Days CRUD, Search, 3 -tier ownership  
Phase 2  Exam Engine Core  5 Days Create, Take, Grade exams  
Phase 3  Integration + Polish  2 Days Bug fixes, UI polish, Testing  
Total MVP Complete  12 Days Deployable product

Senior Developer's Analysis  
Critical Thinking: What Makes This MVP Work  
1. Dependency Chain Analysis  
Before building Question Bank and Exam Engine, we need foundational pieces. Here's the 
dependency chain:  
 
Layer 0 - Infrastructure (Must Have First):  
• Database setup (PostgreSQL via Supabase)  
• Project structure (Node.js + Express + TypeScript)  
• Authentication system (JWT)  
• Basic multi -tenancy middleware  
 
Layer 1 - User Management (Must Have):  
• Users table with role field  
• Tenants table (minimal)  
• Schools table (minimal)  
• Login/logout functionality  
 
Layer 2 - Curriculum Structure (Must Have for Questions):  
• Subjects table  
• Chapters table  
• Topics table (optional for MVP, can be NULL)  
 
Layer 3 - Question Bank (Core MVP):  
• Questions table with 3 -tier ownership  
• Question CRUD APIs  
• Search and filter  
 
Layer 4 - Exam Engine (Core MVP):  
• Exams table  
• Exam sections  
• Exam attempts and responses  
• Auto-grading  
2. MVP Scope Decisions  
DECISION: Start with 4 question types only: MCQ Single, MCQ Multiple, Numerical, True/False. 
Add others post -MVP.  
DECISION: Implement only auto -gradable questions in MVP. Manual grading (subjective) 
comes in Phase 2.  

--- page 4 ---
DECISION: Skip entitlement system complexity. In MVP, all content is tenant -owned or school -
owned. Platform content licensing comes later.  
DECISION: Single exam pattern for MVP: Custom pattern with configurable sections. 
JEE/NEET patterns are presets on top of this.  
DECISION: No real -time features in MVP . Auto -save via HTTP POST every 30 seconds, not 
WebSocket.  
3. Risk Mitigation  
Risk 1: Scope Creep  
• Mitigation: Strict MVP feature list. Everything else goes to 'Phase 2' backlog.  
Risk 2: 3 -Tier Ownership Complexity  
• Mitigation: Implement ownership check as a  single utility function used everywhere.  
Risk 3: Exam Engine Edge Cases  
• Mitigation: Start with happy path. Handle edge cases (browser close, timeout) iteratively.

Technical Decisions for MVP  
Stack Decisions  
Layer  Technology  Rationale  
Database  PostgreSQL (Supabase)  Managed service, built -in auth, storage, realtime if 
needed later  
Backend  Node.js + Express + 
TypeScript  Fast development, good ecosystem  
Frontend  React + Tailwind CSS  Component -based, rapid UI development  
Math Rendering  KaTeX  Fast client -side LaTeX rendering  
Auth JWT (access + refresh)  Stateless, scalable  
API Style  REST  Simple, well -understood, sufficient for MVP  
File Storage  Supabase Storage  Integrated with database  
Architecture Decisions  
ADR -001: 3 -Tier Ownership from Day 1  
Decision: Implement tenant_id + school_id on questions table from the start, even though MVP 
might not use all tiers.  
Rationale: Retrofitting ownership later is painful. The cost of two nullable columns is minimal.  
ADR -002: No WebSocket for MVP  
Decision: Use HTTP POST for auto -save (every 30 seconds) instead of WebSocket.  
Rationale: Simpler infrastructure, easier debugging, sufficient for MVP scale.  
ADR -003: Simplified Grading  
Decision: All -or-nothing grading for MCQ Multiple. No partial marking in MVP.  
Rationale: Partial marking logic is complex. Validate core flow first.  
ADR -004: No Separate Entitlement System  
Decision: Skip platform content and entitlements in MVP. All content is tenant/school owned.  
Rationale: Entitlement system adds significant complexity. MVP proves core value without it.  
ADR -005: PostgreSQL Full -Text Search  
Decision: Use PostgreSQL tsvector + GIN index for question search instead of Elasticsearch.  
Rationale: One less service to manage. PostgreSQL FTS is sufficient for MVP scale (100K 
questions).

Appendix: Quick Referenc e 
A. Question JSONB Formats  
Question Text Format:  
{ html: '<p>What is 2+2?</p>', latex: null, images: [] }  
 
MCQ Options Format:  
[{ id: 'A', text: { html: '3' }, is_correct: false }, { id: 'B', text: { html: '4' }, is_correct: true }, ...]  
 
Correct Answer Formats:  
MCQ Single: { answer: 'B' }  
MCQ Multiple: { answers: ['A', 'C'] }  
Numerical: { value: 42, tolerance: 0.01 }  
True/False: { answer: true }  
B. Exam Status Flow  
DRAFT -> PUBLISHED -> ACTIVE -> COMPLETED  
• DRAFT: Exam being configured, not visible to students  
• PUBLISHED: Exam ready, visible but not yet started  
• ACTIVE: Current time within start -end window  
• COMPLETED: End time passed  
C. Attempt Status Flow  
IN_PROGRESS -> SUBMITTED -> GRADED  
• IN_PROGRESS: Student is taking the exam  
• SUBMITTED: Student submitted or timeout occurred  
• GRADED: Auto -grading completed, score calculated
